Why the explanation?

I'm never happy with simply muddling through a process to get to the end result, so I've been careful in building my sites to gain an insight into what's going on as I do it. And how better to present my new webpage knowledge than in a webpage?

Stage 1: Concept and Content Plan

So I want to make a website, but what exactly will go in it? It's important that the content shapes the design and not vice versa, so my first step was committing my ideas to pen and paper. A simple step, but referring back to these notes continuously throughout the rest of the process proved its worth.

Stage 2: Photoshop Wireframes

I took to Photoshop to quickly sketch out the different areas of the site using simple coloured boxes to represent <div> elements, and named the layers with the classes I would eventually use when coding. As the site is responsive, I made 3 different versions of the wireframes to demonstrate to myself how the site should adapt to the screen size. My design would ultimately change, but having that reference for class names, and a PSD file I could go back and play around with (and build my visual mock-up on) was invaluable.

Later style and practicality would better inform my boxy decisions

Stage 3: Visual Mock-up

Based on the wireframes, I began to add visual elements such as images, text, colours, and backgrounds to provide a goal to code towards. However, I admit I cheated by building the wireframes in code first to see how they'd look with Bootstrap's default styling. Or maybe that's not cheating at all?

Nevertheless my first visual mock-up got scrapped almost completely, its definite estate agent flavour not coinciding with my design goals. The new look is fresher, more original, and appealing to the eye, and it's this second visual mock-up I've been building on since.

I like purple but... this resembled the worst of the Brighton letting agents I've experienced recently

Stage 4: Content and Graphics

As content-writer and selfie-photographer for my own site I had to take some time out to actually write the stuff. Until the text and images were complete I used everyone's favourite fake Latin placeholder text and some fetching photographs of stacks of folders.

I don't purport to be a graphic designer, but a working knowledge of Photoshop helped me to get the graphics I wanted for the site - including designing my own background.

Stage 5: HTML & JavaScript

Leaving the bulk of the styling work aside, I started coding by building the <div> framework using the classes from my wireframes PSD file and the content titles from my content plan. Knowing I would be using Bootstrap's grid system, I added the relevant container, row, and col- classes to create a responsive site before beginning on the true styling.

With the goal of making the site functional before making it pretty, I also coded the JavaScript to allow switching between the content 'tabs'. This was achieved by giving the navigation buttons id attributes then using substrings to find the relevant content to display.

Stage 6: CSS

Some design elements from my visual mock-up were surprisingly easy to lift and apply in CSS, such as the gradient on the header, but others took a bit more fiddling and learning to get right - such as the image caption 'insets' that required different styling for different screen sizes to maintain a satisfying responsive experience.

Now that I'd converted my mock-up into a website, I iterated on various elements of my design to make more informed decisions now that I could test them in the more natural setting of the browser.

I dare say there are more tricks to learn beyond my own brand of code organisation

To write my code I used Sublime Text 3, and quickly became a fan of the Emmet package for shorthand HTML. I'm a shortcut junkie, and these tools scratched an itch when it came to make the process of code writing more efficient.

Stage 7: Hosting

Turns out Adam Lee is a fairly popular name, but the snazzy new .xyz domain was available - hopefully all the snazzier for Google's adoption of it. As I'd been keeping my site backed up and up-to-date on GitHub, hosting it there as a personal portfolio page made sense. And of course, it means all the code is available publicly there now, should you want to see it!

Stage 8: Future considerations

As a fledgling Web Developer, I'm well aware of my limitations. Keeping the site restricted to one HTML document made it easy to edit site-wide elements, such as the header and navigation, but in future the use of PHP would be a more comprehensive solution. Splitting the content to different pages would make for a more logical URL structure and avoid too much content being kept hidden with CSS.

I also hope to develop the JavaScript and jQuery functions of the page to create a more interactive and appealing experience. For example, animating the switch between tabs would create a smoother user experience, as would more dynamic captions and page insets. As a gamer and UX snob, I find the feel of a website or user interface to be a driving factor in how long I'm willing to spend using it. Identifying exactly what makes these experiences so satisfying is one of the most exciting aspects of web development to me.

As I continue to learn more, I intend that the site grow with me as a demonstration of my abilities.
